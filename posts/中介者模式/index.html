中介者模式

中介者模式

March 13, 2020 字数 1193 极速模式

前言

当看到一个类有很多复杂的对象引用和相互调用时,导致代码变得很乱,这个时候就可以使用中介者模式去解决这个问题,比较典型的例子就是MVC。

1.简单说明中介者模式

中介者模式主要是解决对象之间相互引用和调用,导致结构很混乱很复杂。比如游戏引擎模块之间一定是有超级复杂的相互关联问题,这里使用中介者模式来模拟一下如何解决游戏UI模块和Sound模块复杂关系,通过这个设计模式来尽量的做到解耦,使代码之间的调用看起来清晰,具体代码如下:

    public abstract class Mediator
    {
 
        //中介者记录各具体同事类的引用, 用于交互,也可以是集合类记录,如 List, Map
        protected  IGameModule uiGameModule;
        protected  IGameModule soundGameModule;
 
        public IGameModule GetUIGameModule()
        {
            return uiGameModule;
        }
 
        public void SetUIGameModule(IGameModule uiGameModule)
        {
            this.uiGameModule = uiGameModule;
        }
 
        public IGameModule GetSoundGameModule()
        {
            return soundGameModule;
        }
 
        public void SetSoundGameModule(IGameModule soundGameModule)
        {
            this.soundGameModule = soundGameModule;
        }
 
        public abstract void SendMessageToUIGameModule(String message);
 
        public abstract void SendMessageToSoundGameModule(String message);
    }
 
    public class UnityMediator : Mediator
    {  // 具体中介者一般只有一个, 即通用中介者
        public override void SendMessageToUIGameModule(String message)
        {
            uiGameModule.ReceiveMessage(message);
        }
 
        public override void SendMessageToSoundGameModule(String message)
        {
            soundGameModule.ReceiveMessage(message);
        }
    }
 
    public abstract class IGameModule
    {
        protected Mediator mediator;
 
        public IGameModule(Mediator mediator)
        {
            this.mediator = mediator;
        }
 
        public abstract void ReceiveMessage(String message);
 
        public abstract void SendMessage(String message);
 
    }
 
    public class UIGameModule : IGameModule
    {
        public UIGameModule(Mediator mediator) :base(mediator)
        {
 
        }
 
        public override void ReceiveMessage(String message)
        {
            Console.WriteLine("UI模块收到消息: " + message);
        }
 
        public override void SendMessage(String message)
        {
            mediator.SendMessageToSoundGameModule(message);
        }
    }
 
    public class SoundGameModule : IGameModule
    {
        public SoundGameModule(Mediator mediator) : base(mediator)
        {
 
        }
 
        public override void ReceiveMessage(String message)
        {
            Console.WriteLine("Sound模块收到消息: " + message);
        }
 
        public override void SendMessage(String message)
        {
            mediator.SendMessageToUIGameModule(message);
        }
    }

其实通过反射可以实现通用的中介者模式,但是反射的性能指定是不好的,接下通过Unity的SendMessage给大家说明一下可以通用的中介者模式应该这么设计。

2.Unity的SendMessage函数

使用过SendMessage函数的人都知道,在任意继承MonoBehaviour的脚本都可以使用SendMessage调用其他继承MonoBehaviour脚本的函数,此函数设计就可使用中介者模式,当然也可以直接找到对象然后去告诉它具体要干什么事情,具体代码如下:

UIForm uiForm = transform.Find("AboutForm").GetComponent<UIForm>();
uiForm.OnPause();

上面代码相当于告诉AboutForm界面应该要执行暂停操作,如果有大量类似代码会导致对象之间会存在不必要的相互引用、调用, 从而使其耦合。这时就可改成中介者模式,对象之间的相互调用全部在中介者对象里执行,接下来尝试实现SendMessage函数可能的原理,具体代码如下:

    public interface IMediator
    {
        void SendMessage(string MethodName, GameObject gameObject, object[] parameters);
    }
 
    public class UnityMediator : IMediator
    {
        public void SendMessage(string MethodName, GameObject gameObject,object[] parameters)
        {
            MethodInfo methodInfo = gameObject.GetType().GetMethod(MethodName);
            methodInfo?.Invoke(gameObject, parameters);
        }
    }
 
    public class Component : UnityEngine.Object
    {
        private readonly UnityMediator unityMediator = new UnityMediator();
 
        public void SendMessage(string MethodName, GameObject gameObject, object[] parameters)
        {
            unityMediator?.SendMessage(MethodName,gameObject,parameters);
        }
    }

因为是可复用的中介者模式,需要告诉它调用那个脚本下的具体函数,然后它会帮你完成对象之间相互引用的问题。如果对象之间存在比较复杂的引用关系,导致它们之间的依赖关系结构混乱而且难以复用,切记切记！！！使用此设计模式就很好的解决了问题。

总结

1.中介模式的优缺点:

优点:1、降低了类的复杂度程度,从一对多转化成了一对一。 2、各个类之间的解耦。 3、符合迪米特原则。
缺点:1、中介者类会比较庞大,矛盾集中在中介者类里。2、中介者可能会很复杂导致难以维护。

2.中介者模式使用场景:

1.游戏引擎模块之间相互引用和调用时。
