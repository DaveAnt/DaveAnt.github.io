前言

初探篇分析了重写配置模块的思路,按照之前分析的思路尝试重写配置模块,然后为GF配置模块添加xlsx、xls辅助器做准备。

1.配置模块重写

之前问过作者为什么不用dynamic去实现配置模块,作者说框架比较早了当时的解决方案就写成这样了,确实GF框架很早之前就出来了,而且使用dynamic的环境是.net4.0以上,随着时代和技术的发展,到如今dynamic可以满足使用条件了,就让这篇文章来重写Config模块吧!首先需要把ConfigData结构体改成以下所示:

	private struct ConfigData
	{
	    private readonly dynamic m_Value;
	    private readonly Type m_Type;

	    public ConfigData(dynamic m_Value, Type m_Type)
	    {
	        this.m_Value = m_Value;
	        this.m_Type = m_Type;
	    }

	    public dynamic Value
	    {
	        get
	        {
	            return m_Value;
	        }
	    }

	    public Type Type
	    {
	        get
	        {
	            return m_Type;
	        }
	    }

	}

这里用class还是struct看个人需求,在初探篇里面已经说清楚了,如果数据比较少但是频繁取用的话建议改成class,由测试结果而言还是比较倾向使用struct的,ConfigData数据改变以后就需要调整DefaultConfigHelper里AddConfig的代码了,具体修改方案如下图所示:

    protected bool AddConfig(string configName, string configValue)
    {
        dynamic m_Value = null;
        Type m_Type = null;

        int intValue = 0;
        float floatValue = 0f;
        bool boolValue = false;

        if (bool.TryParse(configValue, out boolValue))
        {
            m_Value = boolValue;
            m_Type = typeof(bool);
        }
        else if (int.TryParse(configValue, out intValue))
        {
            m_Value = intValue;
            m_Type = typeof(int);
        }
        else if (float.TryParse(configValue, out floatValue))
        {
            m_Value = floatValue;
            m_Type = typeof(float);
        }
        else
        {
            m_Value = configValue;
            m_Type = typeof(string);
        }
            
        return m_ConfigManager.AddConfig(configName, m_Value ,m_Type);
    }

对于dynamic赋值和用dynamic赋值都不需要进行拆箱和装箱的操作,左右类型不一致就会抛出异常,所以就需要用try-catch捕获代码可能出现的异常,之前模块中GetBool、GetInt、GetFloat等等接口换成GetValue和GetType的两种方法就足够使用。

	public dynamic GetValue(string configName)
	{
	    ConfigData configData;
	    if (!m_ConfigDatas.TryGetValue(configName,out configData))
	    {
	        throw new GameFrameworkException(Utility.Text.Format("Config name '{0}' is not exist.", configName));
	    }

	    return configData.Value;
	}

至于其他细节问题需要对应修改的自行解决~

2.修改后出现的问题

经过一系列猛如虎的操作,如果出现报错了,各位不要惊慌。辛好提前踩过坑了(踩坑的时间确实有点长~),这里就直接说一下可能会遇到的问题。 重新生成解决方案时,发现Unity中还是报了缺少Microsoft.CSharp的库,这时第一个想法就是Unity不能使用dynamic,这样真的是太失败了,先分析一下为什么还是会报错？解决方案的错误已经都解决了,这里报错可能是Unity的.Net环境比较低导致的,所以到PlyerSettings里去查看一下是不是有选择.Net环境的选项(是的,这个操作困扰了我4个多小时…),可以看到以下选择项(Api Compatibility Level)。 可能不知道博主找到这个界面时,当时表情有多猥琐(换你干瞪眼半天,结果突然出现解决方案,应该和我相比好不到那里去~)。 这里修改成4.x以后还是打包测试一下看看有没有具体的问题,在下测试了一下是没有任何问题的。
